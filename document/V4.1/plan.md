# [Stock_Program_V4.1] 아키텍처 및 소프트웨어 공학 기반 추가 패치 계획서 (최종 개정판)

## 1. 프로젝트 개요 및 목표

* **목표:** V4 시스템의 아키텍처적 결함을 수정하고 최적화합니다. 구체적으로 1) V3 레거시의 정교한 비즈니스 룰(P2/P3) 완벽 복원, 2) 전략별 UI 렌더링 분리, 3) 코스피/코스닥 통합 스캔 파이프라인 적용, 4) 무의미한 섹터 집계 로직 제거를 수행합니다.
* **설계 방향:** 비즈니스 로직은 검증된 V3의 룰을 채택하여 신뢰성을 확보하고, 데이터 파이프라인은 V4의 통합 병합 방식을 유지하여 스레딩 효율을 극대화합니다. 모든 과정은 `rule4.md`의 객체지향 설계(SOLID) 및 엄격한 MVC 아키텍처 지침을 준수합니다.

---

## 2. 핵심 설계 및 소프트웨어 공학 원칙 (Injection Rules)

### A. 프로세스 모델 및 방법론 설계

* **프로세스 모델: 반복적 애자일 (Iterative Agile) 모델**
* **선정 이유:** 기존 V4 인프라 위에서 비즈니스 로직 복원, 불필요한 로직 제거, 파이프라인 통합, UI 분리라는 서로 다른 성격의 4가지 작업을 수행해야 합니다. 폭포수 방식보다 각 문제를 짧은 스프린트 단위로 독립적으로 해결하고 통합하는 애자일 방식이 시스템 불안정성을 최소화합니다.


* **적용 방법론: AI-TDD (테스트 주도 프롬프팅)**
* **방법론 적용:** V3의 복잡한 P2(6단계 필터) 및 P3 로직을 `Strategy` 클래스에 복원할 때, 기존 V4 테스트 코드(`test_strategy.py`)를 먼저 수정하여 V3의 세부 조건(거래대금, 이격도 등)을 검증하는 TDD 방식을 강제합니다. 이를 통해 로직 이식 과정에서의 무결성을 보장합니다.



### B. 아키텍처 및 디자인 패턴 선정

* **매크로 아키텍처: 엄격한 MVC (Model-View-Controller) 아키텍처 유지**
* **적용:** 로직(Model)은 V3의 정밀함을 가져오되, 데이터 병합 제어는 Controller(`utils.py`, `market_scanner.py`)에서 수행하고, P1/P2/P3 분리 렌더링은 View(`app.py`, `formatters.py`)에서 전담하도록 계층 간 역할을 엄격히 분리합니다.


* **마이크로 디자인 패턴 (GoF): 전략 패턴(Strategy)의 심화 및 파사드(Facade) 패턴**
* **전략 패턴:** `SectorAnalysisStrategy` 인터페이스 하위의 `P2Strategy`, `P3Strategy` 내부에 V3의 상세 조건들을 캡슐화하여 개방-폐쇄 원칙(OCP)을 준수합니다.
* **파사드 패턴:** `load_tickers` 함수에 파사드 패턴을 적용하여, 클라이언트(Controller)는 내부적으로 KOSPI와 KOSDAQ이 어떻게 병합되는지 알 필요 없이 단일 인터페이스로 통합된 전체 시장 데이터를 제공받도록 설계합니다.



### C. 요구사항 분할(PBS) 및 모델링 전략

전체 패치 작업을 기능적 응집도에 따라 4개의 원자적(Atomic) 단위로 분할합니다.

* **[PBS-4.1.0] 비즈니스 로직 복원 (Model):** V3의 정밀한 P2(수급 주도주 6단계 필터, 상태 분류) 및 P3(급등 전조 텍스트 산출) 비즈니스 룰을 V4의 `Strategy` 클래스에 이식합니다.
* **[PBS-4.1.1] 기능 제거 및 응집도 향상:** 데이터 소스에 존재하지 않는 무의미한 섹터 집계 로직을 전 계층(Model, Controller, View)에서 완전히 도려냅니다.
* **[PBS-4.1.2] 통합 스캔 파이프라인 (Controller):** KOSPI와 KOSDAQ 데이터를 각각 호출한 뒤 `pd.concat`으로 병합하여 `MarketScanner`에 단일 DataFrame으로 주입하는 효율적인 스레딩 구조를 구현합니다.
* **[PBS-4.1.3] 다중 뷰(View) 렌더링 (Presentation):** 결과를 P1, P2, P3 전용 포맷터로 해체하고, UI 상에서 3개의 독립된 섹션으로 렌더링합니다.

### D. 모듈화 및 객체지향 설계 제약 (Modularity & OOP)

* **결합도와 응집도 제어:**
* **Data Coupling 지향:** V3 로직 복원 시, 전략 클래스는 오직 `Pandas DataFrame`만 입력받아 검증 결과가 담긴 `Dict` 객체만 반환하도록 제한하여 모듈 간 결합도를 최하로 유지합니다.
* **Functional Cohesion 극대화:** 뷰(View) 계층에서 하나의 포맷터가 모든 전략을 처리하던 구조를 해체하고, 각 전략 전용 포맷터(`format_p1`, `format_p2`, `format_p3`)를 생성하여 단일 책임 원칙(SRP)을 준수합니다.


* **SOLID 원칙 검증 기준:**
* UI 계층은 V3 룰이 어떻게 계산되는지 전혀 알지 못해야 하며(DIP 준수), 오직 Controller가 전달한 최종 결과 데이터 구조에만 의존하여 화면을 그리도록 설계 검증을 수행합니다.



### E. UI/UX 제약 및 품질 보증(QA) 계획

* **UI/UX 원칙:**
* **멘탈 모델 최적화:** KOSPI와 KOSDAQ이 통합 스캔된다는 점을 직관적으로 알리기 위해, 시장 선택 UI를 단일 선택 상자에서 콤보박스 형태의 다중 선택(Multi-select) 위젯으로 강제 변경합니다.
* **즉각적 피드백:** 두 시장 통합으로 분석 대상이 약 350개로 늘어나므로, `st.progress`와 `st.spinner`를 통한 실시간 진행률 표시를 의무화하여 시스템 멈춤에 대한 사용자 불안을 해소합니다.


* **품질 측정 및 검증 (QA):**
* **순환 복잡도 (Cyclomatic Complexity):** V3의 6단계 P2 조건이 하나의 메서드 안에서 복잡도를 높이지 않도록, 조건 검증(가격, 유동성, 이격도 등)을 개별 헬퍼 메서드로 분리하여 함수당 순환 복잡도를 10 이하로 엄격히 유지합니다.
* **재사용성 (Reusability):** UI 분리 시 공통적으로 사용되는 금액/퍼센트 변환 로직은 별도의 유틸리티 함수로 추상화하여 중복 코드를 방지합니다.
