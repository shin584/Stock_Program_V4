[Stock_Program_Hotfix_V4.2] 아키텍처 기반 고도화 계획서 (개정판)
1. 역할 및 목표
역할: 소프트웨어 공학에 기반한 시스템 아키텍트 및 PM.

목표: 스캐너의 데이터 유실 결함을 수정하고, P2 전략의 논리적 모순을 디버깅하며, P1 뷰 필터 적용 및 메인 UI 리팩토링을 통해 시스템의 무결성과 사용성을 극대화합니다.

2. 영역별 공학 원칙 적용 및 설계
A. 프로세스 모델 및 방법론 설계
프로세스 모델: 버그 픽스 중심의 칸반(Kanban) 기반 애자일

선정 이유: 발견된 4개의 이슈는 상호 의존성이 낮습니다. 결함을 즉각 식별하고 각 계층(Model, View, Controller)별로 독립적으로 수정 및 통합하는 칸반 방식이 가장 빠르고 안전합니다.

적용 방법론: 디버깅 주도 설계 (Debugging-Driven Development)

적용 지점: P2 모듈의 이격도 변수가 5.0 허들을 넘지 못한다는 모순을 발견했으므로, 과열 판단용 '5일선 이격도'와 단계 판별용 '20일선 이격도'의 책임을 분리하는 설계를 강제합니다.

B. 아키텍처 및 디자인 패턴 선정
매크로 아키텍처: 엄격한 MVC 파이프라인 유지

설계: 누락된 25개 종목을 처리하기 위해 Controller(MarketScanner)의 반환 구조를 훼손하지 않습니다. 대신, Controller가 skipped (조건 미달) 상태를 명시적으로 추적하도록 데이터 흐름(Data Flow)을 보완합니다.

마이크로 디자인 패턴: 뷰 필터 패턴 (View Filter) 및 템플릿 메서드 분리

P1 뷰 필터: formatters.py 내부에서 results.head(5)를 적용하여 UI의 렌더링 책임을 캡슐화합니다.

C. 요구사항 분할(PBS) 및 모델링 전략
분석 결과를 바탕으로 PBS를 더 정밀하게 타겟팅합니다.

[PBS-4.2.1] Controller 무결성 복구 (시장 스캐너 보존성): market_scanner.py 수정. result가 None인 종목(조건 미달 종목)을 추적하는 skipped 카운터를 추가하거나, 로그로 남겨 200개 종목의 행방을 사용자에게 투명하게 알리도록 수정.

[PBS-4.2.2] Model 비즈니스 로직 교정 (P2 추세확정): strategy.py 수정. V3의 원래 의도대로 과열 탈락은 5일선 이격도(5% 기준)로 처리하고, 상태(초기/추세) 판별은 20일선 이격도를 별도로 계산하여(예: 5% 기준) 논리적 모순을 해결.

[PBS-4.2.3] View 프레젠테이션 제한 (P1 Top 5): formatters.py 수정. format_p1 함수에 head(5) 데이터 슬라이싱 로직 추가.

[PBS-4.2.4] UI 컴포넌트 재배치 (멘탈 모델 최적화): app.py 수정. StreamlitUI.click_scan 메서드를 삭제하고, main() 함수 내부의 메인 화면 영역(타이틀 하단)에 st.button을 직접 배치하여 직관성을 극대화.

D. 모듈화 및 객체지향 설계 제약 (Modularity & OOP)
단일 책임 원칙 (SRP) 준수:

P2 전략에서 _calculate_ma5_disparity는 오직 '과열 여부'만 판단하도록 하고, 단계 산출을 위한 이격도 계산은 별도의 메서드로 분리하여 함수의 책임을 나눕니다.

결합도(Coupling) 제어:

P1 Top 5 표출을 위해 Model에서 데이터를 자르면, 다른 모듈이 P1 전체 데이터를 활용할 가능성(개방-폐쇄 원칙)을 막게 됩니다. 반드시 View(formatters.py) 계층에서만 화면 표출용으로 자르도록 강제합니다.

E. UI/UX 제약 및 품질 보증(QA) 계획
UX/UI 원칙 (가시성 및 피드백):

메인 화면의 넓은 공간에 '스캔 시작' 버튼을 배치하여 콜투액션(Call To Action)을 명확히 합니다.

200개 종목 스캔 완료 후 요약 메트릭에 "조건 미달" (Skipped) 수치를 추가로 표출하여, 사용자가 '내 종목 25개는 어디로 갔지?'라는 의문을 갖지 않도록 멘탈 모델을 일치시킵니다.
